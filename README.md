# dev-skills

Единый источник правды для моих Agent Skills (правил для AI-агентов).
Каждое правило — отдельная папка с `SKILL.md`. Формат общий для **Claude Code** и **opencode**.

## Как это работает

`~/.claude/skills/` читают оба тула:

- **Claude Code** — своя штатная папка скилов;
- **opencode** — среди прочих локаций сканирует и `~/.claude/skills/*/SKILL.md` (формат идентичен).

Поэтому один симлинк из этого репо в `~/.claude/skills/<name>` подключает правило сразу к обоим.

```
~/Projects/dev-skills/pr-review-comments  ──симлинк──▶  ~/.claude/skills/pr-review-comments
                                                         ▲            ▲
                                                    Claude Code    opencode
```

## Установка

```sh
./install.sh
```

Скрипт проходит по всем папкам правил в репо и создаёт симлинк на каждую в `~/.claude/skills/`.
Идемпотентен: повторный запуск обновляет ссылки и пропускает уже корректные.
`--dry-run` — показать, что будет сделано, без изменений.

## Добавить новое правило

1. Создай папку `<имя-правила>/` с `SKILL.md` (frontmatter: `name`, `description`).
2. `./install.sh` — подключит.
3. Коммит.

## Переносимость в opencode

Правило работает в opencode без изменений, **если не использует Anthropic-специфичные вызовы**.
Опирайся на переносимые инструменты (`gh` CLI, MCP-серверы, обычный shell).
Ссылки на слэш-команды Claude (`/code-review` и т.п.) в тексте — это подсказки для человека, не зависимости; они допустимы.

## Общие контракты (`contracts/`)

Когда несколько скилов должны договориться о формате данных, контракт лежит **один раз** в `contracts/` и подключается в каждый скил симлинком из его `references/` (относительный симлинк корректно резолвится даже через внешний симлинк в `~/.claude/skills`). Правишь в одном месте — видят все.

- `contracts/pr-review.contract.md` — структура файла ревью (frontmatter + finding-блоки) + маппинг finding → inline-коммент GitHub. Общий для `pr-review-comments` (пишет) и будущего `pr-comments-publisher` (читает и постит).

## Общее ядро (`shared/`)

Тон-агностичная методология, переиспользуемая несколькими скилами (симлинк из `references/` каждого).

- `shared/review-core.md` — ядро ревью: Principle #0, find-candidates → verification gate, 4-уровневая шкала, анатомия находки и **формат отчёта** (finding-блок, `Overview`, `Other`, сканируемость). Общий для `pr-review-comments` и `self-review`. Шелл-скил добавляет тон, цель, назначение вывода и frontmatter.

## Правила

- `pr-review-comments` — шелл «чужой PR»: тентативный тон, никогда не постит, paste-ready `.md` (ядро + publisher-контракт).
- `self-review` — *(план)* шелл «свой код»: тот же формат/шкала, прямой тон, пунш-лист для себя (ядро, без publisher-контракта).
- `pr-comments-publisher` — *(план)* читает файл ревью и постит комментарии в GitHub от имени пользователя (pending review).
